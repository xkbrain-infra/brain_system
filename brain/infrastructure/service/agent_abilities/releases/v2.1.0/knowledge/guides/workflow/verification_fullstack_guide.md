# Verification 全栈项目补充指南

> 适用于全栈+多技术域交付（前端/后端/微服务/数据管道/流式/AI/基础设施/安全合规）的 S7 Verification 阶段。
> 本文件是 `verification_guide.md` 的补充，针对全栈项目的特殊需求。
>
> - 基础指南: `verification_guide.md`
> - 流程: `/brain/base/workflow/lifecycle/project.yaml#S7`

---

## 总体原则

1. **风险驱动（Risk-based）**: 先保证 P0 核心链路与不可逆风险（资金/权限/数据/合规）100%
2. **分层测试（Test Pyramid）**: 越底层越多、越稳定；E2E 只保留"核心旅程"
3. **契约优先（Contract-first）**: 跨服务/API/事件/数据 Schema 用契约测试兜底，减少联调返工
4. **可观测性即验收（Observability as AC）**: 监控/日志/告警/trace 必须作为验收条件的一部分
5. **门禁一致（Single Source of Truth）**: 统一 defect 分级、退出标准、Go/No-Go 决策机制

## 全栈覆盖矩阵

每个模块至少打勾到位：

| 技术域 | 必测 P0 | 必测 P1 | 常见坑（必须加用例） |
|--------|---------|---------|---------------------|
| Web 前端 | 核心旅程 E2E、表单校验、权限可见性、关键埋点 | 浏览器兼容、可用性、性能（首屏/交互） | 缓存/本地存储、时区/语言、弱网 |
| 移动端 | 安装/升级、核心旅程、登录/支付(如有)、崩溃率 | 机型/系统版本、离线/弱网、权限弹窗 | 前后台切换、推送、热更新回滚 |
| 后端 API/服务 | 契约测试、鉴权/越权、幂等/重试、错误码一致 | 并发、限流/熔断/降级、回归热点接口 | 超时链路、分页/排序边界、数据一致性 |
| 微服务/消息/事件 | 事件 schema、幂等、重复投递、顺序保证（如需要） | 延迟堆积、死信队列、重放演练 | at-least-once 导致重复副作用 |
| 数据管道（ETL/ELT） | 数据口径、对账、血缘、幂等重跑 | 延迟 SLA、质量监控、异常回补 | schema 漂移、空值/重复、分区错位 |
| 流式计算 | Exactly/At-least 语义、checkpoint、backpressure | 长稳、峰值、延迟分布 | 状态膨胀、重启恢复、乱序数据 |
| AI/检索/推荐 | 离线指标 + 线上 A/B 方案、回滚、guardrail | 漂移监控、滥用/提示注入（如 LLM） | 数据泄露、偏差、回归不可解释 |
| 基础设施/发布 | 灰度、回滚、配置开关、灾备（如要求） | 容量/弹性、发布演练、IaC 变更审计 | 配置漂移、secret 管理、依赖版本 |
| 安全/合规 | RBAC、审计日志、敏感数据处理、漏洞扫描 | 渗透/基线、供应链风险、合规检查 | 日志泄露 PII、权限绕过、默认开放 |

## 测试分层（全栈资源不炸的关键）

### A. 单元测试（Dev 主责）

- 覆盖：核心逻辑、边界条件、异常分支
- 目标：快、稳定、PR 必过

### B. 服务/API 测试（QA + Dev）

- 覆盖：主要接口 happy/negative、鉴权、错误码、幂等、限流
- 优点：稳定性高、回归性价比最高（**建议成为回归主力**）

### C. 契约测试（强烈建议）

- 覆盖：API schema、事件 schema、数据表 schema（consumer/provider）
- 目标：减少"联调才发现字段变了/语义变了"

### D. 集成测试（环境级）

- 覆盖：跨服务、第三方依赖、异步链路、消息消费
- 必须包含：超时/重试/降级/熔断的行为验证

### E. 端到端 E2E（少而精）

- 只保留 P0 核心旅程（例如：注册 → 登录 → 下单 → 支付 → 查询 → 退单）
- 原则：E2E 用例数量宁可少，也要稳定可复跑

### F. 非功能（NFR）专项

- 性能（P95/P99、吞吐、长稳）
- 安全（越权、数据泄露、审计）
- 可用性（容灾、回滚、降级）
- 可观测性（监控/告警/日志/trace）

## 全栈执行节奏

### 阶段 1：环境与构建门禁（每天都要过）

- Build 可部署
- 环境健康检查（依赖服务、队列、数据库、配置、证书）
- 冒烟用例（15-30 分钟内完成）

### 阶段 2：功能 + 契约 + 集成（主战场）

- 需求 → 用例 → 执行 → 缺陷 → 回归闭环
- 每天固定一次 Defect Triage（统一定级与修复承诺）

### 阶段 3：回归（锁范围后）

- 变更影响面回归（impact-based regression）
- 历史 bug hotspot 回归

### 阶段 4：NFR 与发布就绪（上线前必须完成）

- 性能/容量/长稳（至少覆盖 P0 链路）
- 安全/审计/合规清单确认
- 监控告警、灰度、回滚演练

## 全栈缺陷管理

### 修复 SLA

| 级别 | SLA |
|------|-----|
| S0 Blocker | 当天必须出修复或降级方案 |
| S1 Critical | 24-48h 内 |
| S2 Major | 进入版本评估（可延后但要记录理由） |
| S3 Minor | 积累到后续迭代 |

## 全栈 Go/No-Go 门禁

### Exit Criteria（QA 测试退出）

- [ ] P0 用例通过率 100%
- [ ] S0/S1 = 0（或有正式降级/规避方案并审批）
- [ ] 回归覆盖变更影响面（有证据：用例/报告/日志）
- [ ] 性能达到 NFR（或已配置限流/降级且验证生效）
- [ ] 安全与权限用例通过（越权、敏感数据、审计日志）
- [ ] 可观测性就绪：关键指标、仪表盘、告警阈值、trace/log 可定位
- [ ] 灰度/回滚方案可执行且演练过（至少 dry-run）

### Release Readiness（上线就绪）

- [ ] 灰度策略（扩量条件、观察窗口、回滚阈值）
- [ ] On-call 值班与升级路径
- [ ] 数据迁移/脚本已演练（如有）
- [ ] Release notes / 变更记录 / 已知问题清单完成

## Verification Package（全栈交付物）

上线评审用的可审计材料：

1. **Test Plan** — 范围/策略/环境/退出标准
2. **Traceability Matrix** — 需求 ↔ 用例 ↔ 缺陷 ↔ 版本
3. **测试报告** — 通过率、覆盖、未测项解释
4. **缺陷报表** — 按 S0-S3、模块、趋势、修复率
5. **NFR 报告** — 性能/安全/长稳/可观测性
6. **发布就绪清单** — 灰度/回滚/监控/值班
7. **已知问题与风险接受** — 谁批准、为什么

## 组织与职责（RACI/DRI）

| 角色 | 职责 |
|------|------|
| QA（DRI） | 测试策略、用例设计、执行与报告、门禁建议 |
| Dev（Owner） | 单测/修复/回归自测、提供 debug 能力（日志/trace） |
| PM（Approver） | 范围锁定、需求口径、风险接受与取舍 |
| SRE/Infra | 环境稳定、发布/回滚、容量与告警 |
| Security/Compliance | 安全与合规门禁签核（需要则必入） |
